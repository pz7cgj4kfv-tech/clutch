## ✅ CE QUI CONVERGE (les deux panels d'accord = on prend)

1. **PIÈGE N°1 — « vous testez le VIDE, pas Clutch ».** 30 personnes non-synchronisées = même Tinder paraît mort.
   → **Le 1er test doit être CHORÉGRAPHIÉ**, assumé comme une simulation sociale réelle.
2. **Zéro bot d'ambiance romantique.** Refus net (Growth + Sécurité). LPD : transparence + finalité du traitement
   centrales → simuler de fausses personnes désirables = mauvaise base de confiance + risque légal. Bots = **Test
   Lab uniquement, étiquetés « BOT TEST »**, jamais dans l'expérience réelle des testeurs.
3. **Forteresse = nécessaire mais NON suffisante.** Moat **interne** (qualité + sécurité : moins de lapins, plus de
   check-ins réussis), **pas** un argument d'acquisition. Elle doit être **active** au test, mais **jamais pitchée**.
4. **Avant le test, le CYCLE HUMAIN COMPLET doit tenir** : ouvrir dispo → voir quelqu'un → envoyer Clutch →
   accepter/refuser → Verrou → check-in (ou fallback) → terminer → feedback. Le graphisme peut être moyen.
5. **Jamais « rien ne se passe ».** Tout blocage = message de raison (neutre côté user, détaillé côté admin QA).
6. **Notifs fiables** (Clutch reçu / accepté / refusé / Verrou) + **blocage / signalement / refus doux** visibles.

## 🗑️ CE QUE JE VIRE (bluff ou hors-sujet POUR CE test de 30 amis)

- **« Modération texte serveur renforcée »** (cité par les deux) → over-engineering pour un test entre amis de
  confiance. Vrai pour le **lancement public**, pas pour la soirée test.
- **Grok / CTO : « Forteresse dynamique GPS + i18n complet + premium stub » comme manques** → se **contredit**
  avec sa propre liste « ne pas faire » ET avec GPT. Consensus réel = **rien de ça n'est requis pour tester**.
- **« NPS après chaque event » (Grok)** → trop corporate pour 30 potes. **Feedback verbatim en vrac** (méthode
  David) capte mieux et plus honnêtement.

## 🎯 DÉCISIONS TRANCHÉES (rangées dans la vision)

### A. La Forteresse reste CACHÉE — on n'affiche que le RESSENTI (décision David, confirmée par les 2 panels)
- La Forteresse / le Cône = **moteur sous le capot** (« truc de geek très puissant »). **On ne le met PAS en avant.**
- Côté utilisateur : **uniquement le RESSENTI**. Quand il se met dans une situation qui **devient impossible**, il
  le **SENT** — sans voir l'algo, les chiffres, ni les mots « forteresse » / « cône » / « D + R ≤ portée(Δt) ».
- **Ce ressenti EXISTE DÉJÀ** = le **cône vivant** (slider vert→rose→violet qui se resserre + tension + boussole
  micro-signal + haptique aux seuils + tick 30 s). Construit comme une **sensation**, pas un dashboard. ✅
- **RESTE** : le **nommer côté user** sans jargon. Pistes (à choisir, jamais imposer) : « ta fenêtre se referme »,
  « ça devient tendu », un **souffle visuel** qui se contracte. → décision design (Mel), 2-3 directions.

### B. Le 1er test = une SOIRÉE CHORÉGRAPHIÉE (« Clutch test night »)
- **Cadre imposé** : 1 soir (ex. jeudi), **2 h précises** (19h30–21h30), **centre Lausanne**, **rayon 2–5 km**.
- **Consigne explicite** au groupe : « **à 19h30, tout le monde ouvre une dispo maintenant** ».
- **Groupe WhatsApp/Telegram fermé** de coordination + relances.
- L'app = **outil de coordination** de cette densité ; on teste le **matching spontané** en conditions réelles.
- Variante : ancrer sur un **mini-event réel** (apéro dans un bar) — mais alors on teste surtout l'**event**, moins
  le spontané. → faire les deux sur 2 soirées différentes.

### C. AVANT la soirée — la checklist « cycle humain qui tient »
- [ ] Profil minimal propre (prénom, âge, photo, intention, genre/recherche).
- [ ] **Notifs fiables** (Clutch reçu **app fermée** = critique).
- [ ] **Aucun écran muet** : chaque blocage affiche une raison (neutre user / détaillée admin).
- [ ] Blocage / signalement / **refus doux** + cooldown actifs et visibles.
- [ ] Consentement GPS clair + lieux publics + jamais de distance exacte (radar = temps).
- [ ] Onboarding 60 s pour non-initiés.
- [ ] Test Lab gardé en admin, bots « BOT TEST » **séparés** de l'expérience réelle.

### D. PLUS TARD (ne pas faire pour le 1er test)
Bots d'ambiance · premium/crédits/boosts · hors Lausanne · Clutch City comme argument testeurs · forteresse GPS
dynamique parfaite · i18n complète (testeurs francophones) · modération texte serveur · graphisme pixel-perfect.
