0 · Vision & Promesse fondatrice

"Présence spontanée vérifiée dans le monde réel."

Clutch n\'est pas un meilleur Tinder. C\'est une nouvelle catégorie. Là où les apps de rencontres créent des cimetières de matchs non-concrétisés, Clutch impose l\'action : un Clutch = un rendez-vous proposé dans les 2h, à réaliser dans les 18h max. Ou ça expire. La friction est volontaire. La spontanéité est architecturale.

Le GPS n\'est pas là pour espionner — il est là pour que tu saches que la personne est vraiment à Lausanne maintenant, pas à Paris en train de swiper. La position est une zone choisie, jamais des coordonnées exactes.

❌ Tinder
Infiniment de profils, aucun RDV. Le chat remplace le vrai contact.
✅ Clutch
Profils disponibles MAINTENANT. Le chat est interdit. RDV dans les 18h ou rien.
❌ Meetic
Planifier un RDV 2 semaines à l'avance après 50 messages.
✅ Clutch
Pas de chat. Un Verrou = un engagement. Le reste est IRL.

ADN produit — ce qui ne changera jamais

VocabulaireClutch / Verrou / Présences. JAMAIS "match", "swipe", "like". Le vocabulaire est la promesse.
Contrainte 18hTout timing respecte cette fenêtre. L'étirer = trahir le concept. Non-négociable.
Femmes au centreElles sont le centre gravitationnel. Sans elles les hommes partent. Gratuité = permanente.
Friction utile"Cette feature rend-elle Clutch plus confortable depuis le canapé ?" Si oui → dark pattern.
Belle énergieL'app doit irradier de la vie sociale positive. "Se rencontrer vraiment" > "trouver un match".

1 · Décisions validées & justifications

Ces décisions sont actées. Elles ne se remettent pas en question sans accord de l\'équipe fondatrice.

✅
Vocabulaire propriétaire : Clutch / Verrou / Présences (jamais "match" ou "swipe")
Un langage propre = une culture défendable. C'est l'équivalent du "tweet" pour Twitter.
✅
Femmes gratuites à vie
Éthique ET stratégique. Sans elles, les hommes partent. Ce n'est pas un choix commercial, c'est une contrainte de design.
✅
Contrainte 18h max sur tous les créneaux
Structurel et non-négociable. L'étirer = trahir le concept. La contrainte crée l'urgence = la valeur.
✅
GPS : zone choisie, pas position live
LPD suisse. Localisation = donnée sensible. On stocke "disponible au centre-ville", jamais les coordonnées exactes en DB.
✅
Score fiabilité public (affiché, pas caché)
Responsabilisation sociale. Les comportements invisibles restent impunis. Ici le lapin coûte publiquement.
✅
Apple IAP CHF 19.90/mois hommes premium
Testé mentalement : 30% commission Apple = CHF 14 net/mois. 1000 premiums = CHF 14k/mois = viable. Gratuit femmes = équilibré.

2 · Bugs détectés — Audit simulation 16.06

7 scénarios testés par 9 agents en parallèle (4 min 33s). Appels HTTP réels contre l\'API Supabase. Score avant fixes : 4/10. Score attendu après : 8/10.

DB-01Trigger clutches référençait colonne "plan" inexistanteCORRIGÉ ✅
Le trigger check_clutch_limit() cherchait NEW.plan mais la colonne s'appelle account_type dans profiles. Résultat : TOUT INSERT dans clutches échouait. Aucun Clutch ne pouvait être envoyé.
→ Trigger recréé avec la bonne colonne. Code JS corrigé (.plan → .account_type). SQL appliqué en DB le 16.06.
DB-02Contrainte rdv_feedbacks rejetait "on_time" et "showed"CORRIGÉ ✅
La contrainte CHECK n'acceptait que "absent". Le code JS envoyait aussi "on_time" et "showed". Quand les premiers vrais users auraient fait un feedback positif, ça aurait échoué silencieusement.
→ ALTER TABLE rdv_feedbacks: contrainte mise à jour pour accepter on_time/showed/absent. SQL appliqué le 16.06.
DB-03RLS UPDATE manquant sur profiles et clutchesCORRIGÉ ✅
Un PATCH anon retournait HTTP 200 au lieu de 401. Sur des tables vides pas de risque concret, mais en production des données auraient pu être modifiées sans authentification.
→ Policies CREATE POLICY profiles_update et clutches_update ajoutées. SQL appliqué le 16.06.
DB-04user_feedbacks lisible sans authentification🔴 OUVERT
La table user_feedbacks est accessible en lecture publique (HTTP 200 avec données). Les feedbacks de "Tafit" incluant son identifiant Supabase sont visibles sans connexion. Fuite de données LPD.
→ ALTER TABLE user_feedbacks ENABLE ROW LEVEL SECURITY + CREATE POLICY feedbacks_select. Inclus dans le SQL du 16.06 — à vérifier que c'est bien appliqué.
DB-05Anti-doublon et self-Clutch non enforced en DB🟡 À FAIRE
La logique "une seule demande par paire" et "on ne peut pas s'envoyer un Clutch à soi-même" est 100% côté JS. Un utilisateur via API peut contourner. Pas critique avec 0 vrais users mais à corriger avant lancement.
→ UNIQUE constraint sur (sender_id, receiver_id) + CHECK constraint sender_id != receiver_id. À appliquer dans Supabase.
DB-06reliability_score calculé côté JS uniquement🟡 À FAIRE
Le score de fiabilité est mis à jour en JS après chaque feedback. Un utilisateur malveillant peut signaler des lapins sans que son score soit affecté s'il bypass le client.
→ Créer une Edge Function Supabase qui calcule le delta de score après INSERT dans rdv_feedbacks et applique la mise à jour sur profiles.
PERF-01Performances API Supabase✅ OK
Tests de charge effectués par l'agent de simulation : temps de réponse 26-76ms en régime normal. Tables vides actuellement (0 vrais users). À remonitorer à partir de 500 users.
→ RAS. Indexer is_available + available_until si les requêtes ralentissent.
GPS-01GPS non stocké en clair — conforme LPD✅ OK
Confirmé par audit : les coordonnées GPS ne sont pas stockées en DB. Seule la "zone de disponibilité choisie" est enregistrée. Le calcul de proximité est 100% côté client.
→ RAS. À maintenir comme invariant dans toutes les futures features GPS.

SQL restant à appliquer dans Supabase

-- Anti-doublon : contraintes DB manquantes ALTER TABLE clutches ADD CONSTRAINT no_self_clutch CHECK (sender_id != receiver_id); CREATE UNIQUE INDEX IF NOT EXISTS clutch_pair_unique ON clutches(sender_id, receiver_id) WHERE status IN ('pending','confirmed','accepted'); -- Vérifier que RLS user_feedbacks est actif : ALTER TABLE user_feedbacks ENABLE ROW LEVEL SECURITY;

A-10Double feedback caché révélé 3h après le RDV
✅ ImplémentéClaude

Après un RDV, les deux parties reçoivent une notification 3h après. Chacun répond sans voir la réponse de l'autre : À l'heure / Venu·e / Lapin. Les deux réponses sont révélées en même temps seulement quand les deux ont répondu (ou après 24h timeout).

POURQUOI C\'EST CLUTCH
Le délai de 3h réduit les feedbacks vengeance à chaud. Le double révélation symétrique empêche d'influencer la réponse de l'autre. C'est le même mécanisme que les négociations de salaire bilatérales.
COMMENT ÇA MARCHE
Stocké dans rdv_feedbacks. Révélé par une Edge Function ou un cron toutes les 15min. Score mis à jour automatiquement.
⚠ RISQUES : La contrainte DB outcome n'acceptait que "absent" — corrigé en session 16.06 pour accepter on_time/showed/absent.

B-01Mission du Jour
🔜 Phase 2GPT

Table missions en DB avec un cron quotidien. Badge assigné après feedback positif du RDV de ce jour. Visibilité boostée = apparaît en premier dans Présences.

B-02File d'attente intelligente pour les femmes
🔜 Phase 2GPT

Au lieu de recevoir toutes les demandes de Clutch en vrac, les femmes reçoivent une sélection de 3 meilleurs candidats, renouvelée toutes les heures. L'algorithme classe par : score de fiabilité, compatibilité d'activité du moment, diversité de profils (pas 3 développeurs). Les demandes non-sélectionnées sont mises en file et pourraient être proposées plus tard.

POURQUOI C\'EST CLUTCH
Supprime la surcharge cognitive. "J'ai 47 demandes" est paralysant. "Voici 3 personnes intéressantes maintenant" est activable. Améliore la perception de qualité de l'app.
COMMENT ÇA MARCHE
Algo de scoring côté serveur à construire. Renouvellement horaire via cron. Les hommes ne savent pas qu'ils sont en file d'attente.
⚠ RISQUES : Complexité algorithmique. Risque que les hommes patientent trop longtemps. Solution : timeout 2h si non-sélectionné → rejeté proprement.

6 · Fossé défensif — ce que les concurrents ne peuvent pas copier

Quand Clutch marche, les géants voudront copier. Ce qui est non-copiable sans repartir de zéro :

🔒
Réputation publique réelle
Le score de fiabilité est basé sur des comportements réels vérifiés. Il faut des années de data pour le construire. Impossible à copier à froid.
📖
Culture & vocabulaire propriétaires
Clutch / Verrou / Présences. Un langage = une tribu. Les utilisateurs partagent ces mots entre eux. Ça ne se copie pas.
🏙
Réseau local Lausanne-first
L'effet réseau local est défensif. EPFL + UNIL + bars partenaires = masse critique locale difficile à déloger.
🕸
Historique social et connexions
Après 10 rencontres : "Vous avez 4 personnes en commun." Ce graphe social local ne peut pas être recréé sans plusieurs années d'usage.
⏱️
Contrainte 18h architecturale
Ce n'est pas un feature, c'est la définition du produit. Le copier = faire Clutch, pas un clone de Tinder.
⚖️
Système de responsabilisation
Le double feedback + le score public + l'histoire de fiabilité forment un écosystème de confiance qui prend des années à construire.
